# Архитектура и Структура проекта

Бот спроектирован по модульному принципу, что обеспечивает удобное масштабирование и изоляцию логики. Взаимодействие компонентов выстроено через слои (роутеры, сервисы, репозитории базы данных).

## Структура директорий

```
business_card_bot/
├── app/
│   ├── bot/                # Телеграм бот: роутеры, хэндлеры, клавиатуры, тексты
│   │   ├── routers/        # Обработчики сообщений и коллбеков (aiogram Routers)
│   │   ├── keyboards/      # Клавиатуры (Reply, Inline)
│   │   ├── texts/          # Тексты бота (Markdown файлы)
│   │   ├── callbacks.py    # CallbackData классы для Inline кнопок
│   │   ├── middlewares.py  # Aiogram Middlewares (например, для сессии БД)
│   │   ├── text_catalog.py # Парсер текстов из Markdown файлов
│   │   └── utils.py        # Утилитарные функции для бота
│   ├── db/                 # Слой работы с базой данных
│   │   ├── models.py       # Модели SQLAlchemy
│   │   ├── repositories.py # Классы-репозитории для CRUD операций с БД
│   │   ├── session.py      # Настройка асинхронного движка и сессий
│   │   └── base.py         # Базовый класс для моделей
│   ├── services/           # Бизнес-логика (независимая от aiogram и БД)
│   ├── migrations/         # Файлы миграций Alembic
│   ├── config.py           # Настройки pydantic-settings (парсинг .env)
│   └── main.py             # Точка входа, инициализация бота, Redis, Postgres
├── scripts/                # Скрипты для разработки (dev.ps1)
├── tests/                  # Pytest тесты
├── docker-compose*.yml     # Конфигурация Docker
└── pyproject.toml          # Зависимости и метаданные проекта
```

## Взаимодействие компонентов (Архитектура)

Согласно графу зависимостей, основные "сообщества" (логические блоки) в проекте и их взаимодействия:

1. **`routers-case`** и **`bot-callback`**:
   Это сердце Telegram-интерфейса. Роутеры перехватывают апдейты (сообщения, нажатия кнопок) и используют `CallbackData` для извлечения данных из нажатий Inline-кнопок.
   *Высокая связанность* (high coupling) между этими модулями ожидаема, так как роутерам необходимо создавать и парсить callback-вызовы.

2. **`keyboards-keyboard`**:
   Генераторы клавиатур тесно связаны с `bot-callback`, так как каждая Inline-кнопка содержит экземпляр класса `CallbackData`. Клавиатуры вызываются внутри роутеров при формировании ответа.

3. **`db-case` (Database Repository)**:
   Роутеры (`routers-case`) взаимодействуют с базой данных не напрямую через SQLAlchemy, а через **Паттерн Репозиторий** (`repositories.py`). Это позволяет скрыть SQL-запросы от роутеров. Роутер вызывает метод репозитория (например, `get_case`), репозиторий возвращает объект.

4. **`app-settings` (config.py)**:
   Используется повсеместно (база данных, роутеры, сервисы) для получения токенов, ID администраторов, ссылок и других констант из `.env`.

5. **`tests-text`**:
   Слой тестирования активно ссылается на роутеры и клавиатуры, чтобы гарантировать корректность генерации контента.

## Жизненный цикл обработки сообщения

1. Telegram отправляет Update на сервер.
2. Aiogram `Dispatcher` принимает Update.
3. Отрабатывают `Middlewares` (например, `DbSessionMiddleware` открывает сессию базы данных и кладет её в `kwargs` обработчика).
4. Update проходит через фильтры (например, проверка, что пользователь является админом по `ADMIN_IDS`).
5. Update попадает в `Router`, где вызывается нужный хэндлер.
6. Хэндлер:
   - Получает данные (например, `session` БД).
   - Вызывает `CaseRepository` для получения информации из базы.
   - Обращается к `text_catalog` за нужным текстом.
   - Формирует клавиатуру.
   - Отправляет ответ пользователю (`message.answer` или `callback.message.edit_text`).
7. `Middleware` закрывает сессию БД.
