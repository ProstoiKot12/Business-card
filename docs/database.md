# База данных и Миграции

В проекте используется реляционная база данных **PostgreSQL 16**.
Для взаимодействия с БД из Python используется асинхронный ORM **SQLAlchemy 2** в связке с драйвером `asyncpg`.
Управление структурой (миграциями) осуществляется через **Alembic**.

## Модели данных

Модели определены в файле `app/db/models.py`.

### Таблица `cases` (Кейсы/Портфолио)

Основная бизнес-сущность проекта — это "Кейс" портфолио.
Модель `Case` наследуется от `Base` и `TimestampMixin` (добавляет поля `created_at` и `updated_at`).

**Основные поля `Case`:**
- `id` (Integer): Первичный ключ.
- `order` (Integer): Индекс сортировки (для изменения порядка кейсов). Индексировано.
- `emoji` (String): Эмодзи кейса (например, 🚀, 💻).
- `title` (String): Название проекта.
- `type` (String): Тип работы (например, "Бот", "Веб-приложение").
- `task` (Text): Описание поставленной задачи.
- `solution` (Text): Описание решения.
- `result` (Text): Достигнутый результат.
- `stack` (JSONB): Список технологий (массив строк).
- `media_group` (JSONB): Список `file_id` фотографий в Telegram для отправки MediaGroup.
- `bot_link` (String, nullable): Ссылка на работающий проект.
- `extra_link` (String, nullable): Дополнительная ссылка (например, на GitHub).
- `is_visible` (Boolean): Флаг видимости (включен по умолчанию). Индексировано.

## Использование Паттерна Репозиторий

Вся логика работы с базой данных инкапсулирована в `app/db/repositories.py`.
Вместо того чтобы писать запросы прямо в Aiogram-хэндлерах, вы используете класс репозитория.

Пример использования:
```python
# В aiogram хэндлере:
async def get_all_cases_handler(message: Message, session: AsyncSession):
    repo = CaseRepository(session)
    cases = await repo.get_visible_cases()
    # Дальнейшая работа с полученными кейсами
```

## Работа с миграциями (Alembic)

Alembic отслеживает изменения в моделях `app/db/models.py` и генерирует файлы миграций в папке `app/migrations/versions`.

### Создание новой миграции

Если вы изменили, добавили или удалили поля в `models.py`, вам необходимо создать миграцию.
В Dev-окружении (при активированном `.venv`):

```bash
alembic revision --autogenerate -m "Описание изменений, например: add views_count field"
```
Убедитесь, что Alembic корректно сгенерировал код в новом файле миграции.

### Применение миграций

Миграции автоматически применяются при запуске `scripts/dev.ps1`, если есть новые файлы.
Для ручного применения миграций до последней версии выполните:

```bash
alembic upgrade head
```

Отмена последней миграции:
```bash
alembic downgrade -1
```

## Начальное заполнение базы (Seeds)

В проекте есть скрипт для заполнения базы демо-данными (seeds), чтобы вам не пришлось настраивать пустого бота.
Обычно он вызывается при инициализации, если база пуста (смотрите логику в `main.py` или `app/db/seeds.py`).
